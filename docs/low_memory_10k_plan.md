# 1万标样 + 2GB 内存：检索架构方案

## 目标
- 在约 10,000 张标样规模下保持可用检索速度
- 在 2GB 内存约束下避免 OOM
- 尽量保持当前接口形态（`/search`）不变

## 现状风险
- 在线提特征（逐图）在大规模下启动慢、内存波动大
- 单进程加载模型后可用内存空间很小
- 全量点积扫描 + 多视图重排会导致请求时延明显上升
- 多 worker 会放大内存占用，2GB 下易触发 OOM

## 总体策略
1. **离线化**：标样特征全部离线计算并落盘  
2. **索引化**：用 ANN 索引先召回，再小范围精排  
3. **轻量化**：降维/量化特征，压缩内存与 I/O  
4. **限流化**：在线流程只保留必要步骤，严格控制重排成本

## 推荐架构
### A. 离线构建（批处理）
- 脚本：`build_index.py`
- 输入：`data/standard_samples/*`
- 输出：
  - `artifacts/features.f16.npy`（或量化特征）
  - `artifacts/meta.json`（图片名、款号映射）
  - `artifacts/faiss.index`（IVF/HNSW）
- 建议：
  - 原始 512 维先做 PCA 到 128 或 256 维
  - 特征用 `float16` 存储，必要时再做 PQ/INT8

### B. 在线服务（API）
- 启动阶段只做：
  - 加载索引文件与元数据
  - 不再遍历标样提特征
- 请求阶段：
  1) 查询图提一次特征  
  2) ANN 召回 topN（如 50~200）  
  3) 对 topN 做款号聚合与轻量重排  
  4) 返回 topK

## 参数建议（2GB 场景）
- `workers=1`
- `query_multicrop=false`（默认关）
- `query_component_views=false`
- `rerank_topn` 控制在 `8~20`
- `candidate_multiplier` 控制在 `4~8`
- 仅在必要时开启 base64 返回

## 内存与性能估算（粗略）
- 10k × 512 × float32 ≈ 20MB（仅向量本体）
- 关键开销不在向量本体，而在：
  - Python 进程 + 模型加载
  - 预处理与中间缓存
  - 多 worker 重复占用
- 因此必须“离线 + 索引 + 单进程”

## 实施步骤
1. 新增离线索引构建脚本（批处理）
2. API 增加“索引模式”加载路径（优先使用索引）
3. 灰度：对比旧方案准确率与时延
4. 稳定后将在线逐图特征构建改为仅调试用途

## 验收标准
- 冷启动可控（不再逐图提特征）
- 单次 `/search` 平均时延显著下降
- 2GB 机器持续运行不出现 OOM
- 在目标测试集上准确率不低于基线可接受范围

## 风险与回退
- 风险：降维/量化可能带来准确率下降
- 回退：
  - 提高召回 topN
  - 提高精排权重
  - 使用更高维 PCA 或关闭强量化

## 1万标样：最优硬件方案（高性能路线）

## 目标
- 在保证准确率的前提下，将首包时延和尾延迟压到更低
- 支持更高并发和更复杂在线重排策略

## 推荐硬件
### 方案 A（性价比高）
- CPU：8~16 vCPU
- 内存：32GB
- GPU：单张中端推理卡（如 16GB 显存级别）
- 磁盘：NVMe SSD（>1GB/s）
- 网络：公网带宽稳定，建议配合 CDN/WAF

### 方案 B（高性能）
- CPU：16~32 vCPU
- 内存：64GB
- GPU：单张高性能推理卡（24GB+ 显存）
- 磁盘：NVMe SSD（高 IOPS）

## 架构建议（高配）
1. **离线索引 + 在线 GPU 提特征**
   - 查询图特征由 GPU 推理
   - 标样特征离线构建，在线只读索引
2. **ANN 两阶段召回**
   - 第一阶段 ANN（大召回）
   - 第二阶段精排（可加更强 reranker）
3. **服务分层**
   - API 网关层（Nginx/Cloudflare）
   - 检索服务层（多副本）
   - 索引与模型文件共享层（对象存储或本地 NVMe）

## 参数建议（高配默认）
- `workers=2~4`（按 CPU 和显存压测决定）
- `query_multicrop=true`
- `query_component_views=true`
- `candidate_multiplier=12~20`
- `rerank_topn=20~80`
- 可开启更强重排模型（跨模态或 Siamese）

## 预期效果（相对低配）
- 冷启动：更快（索引加载+GPU初始化后稳定）
- 稳态请求：延迟更低且抖动更小
- 并发能力：明显提升
- 可用更复杂特征与重排策略而不牺牲吞吐

## 运维建议
1. 全链路指标
   - QPS、P50/P95/P99、错误率、GPU 显存/利用率、CPU/内存
2. 灰度发布
   - 新模型/新索引先灰度 5% 流量
3. 版本化
   - 模型版本、索引版本、配置版本三者独立管理
4. 自动回滚
   - 当 P95 或错误率超过阈值自动回退到上一版本

## 与低内存方案的取舍
- 低内存方案优先“能跑、稳、不 OOM”
- 高配方案优先“更低延迟、更高并发、更高准确率上限”
- 推荐做“双配置运行”：低配保底，高配承载主流量

## 预算-性能对照表（参考）

> 说明：以下为工程规划参考区间，实际表现受网络、图片分辨率、重排复杂度、并发模型、云厂商实例性能波动影响。  
> 指标以“10k 标样、API 检索、已做离线索引”为前提估算。

| 档位 | 典型硬件 | 预估月成本（云主机） | 预估 P95 时延 | 预估稳定 QPS | 适用场景 |
|---|---|---:|---:|---:|---|
| 极简保底 | 2 vCPU / 2GB / 无GPU | 低（~¥100-300） | 1.2s~3.5s | 0.5~2 | 验证、低流量内测 |
| 经济可用 | 4 vCPU / 8GB / 无GPU | 中低（~¥300-800） | 400ms~1.2s | 3~10 | 小规模线上 |
| 性价比主力 | 8 vCPU / 32GB / 单GPU(16GB) | 中高（~¥1500-5000） | 120ms~500ms | 15~60 | 主流业务流量 |
| 高性能 | 16+ vCPU / 64GB / 单GPU(24GB+) | 高（~¥5000+） | 80ms~300ms | 50~150 | 中高并发、低时延要求 |

## 成本优化建议
1. 优先优化索引与缓存，再升级硬件  
2. 高峰期弹性扩容，低峰降配  
3. 对图片做输入尺寸上限控制，减少无效算力开销  
4. 按业务分层：免费/低优先级走低配队列，核心请求走高配队列

## 容量规划方法（建议）
1. 先确定业务目标：目标 QPS、P95、可接受错误率  
2. 压测三组配置（低/中/高）并记录：
   - `QPS vs P95`
   - CPU/内存/显存峰值
   - 错误率与超时率
3. 选择“满足 SLO 的最低成本配置”作为基线  
4. 预留 30%~50% 余量应对峰值流量
